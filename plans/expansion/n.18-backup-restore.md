### §1 — What to Back Up
Numbered list with shell snippets:

1. **Database** — `mysqldump -u phlex -p phlex_db > backup.sql` (or MariaDB equivalent);
   watch history and user profiles are in the DB.
2. **Config files** — `config/` directory (`server.php`, `database.php`, `ffmpeg.php`,
   `backups.php` if present); includes JWT_SECRET and DB credentials.
3. **Metadata images** — `data/metadata/` (poster.jpg, fanart.jpg); can be
   regenerated by re-scanning library but re-downloads from TMDB/TVDB.
4. **Plugin configs** — `data/plugins/` if any custom plugins are installed.
5. **TLS certificates** — `config/ssl/` for self-hosted deployments.
6. **NOT needed** — transcoded cache (regenerated on next transcode), active
   transcode job state (lost on crash, not worth capturing), scan cache
   (regenerated on rescan).

### §2 — Backup Methods

#### 2.1 Manual Backup (CLI)
```bash
# Database
mysqldump -u phlex -p phlex_db > phlex-db-$(date +%Y%m%d).sql

# Full tarball of everything that matters
tar czf phlex-backup-$(date +%Y%m%d).tar.gz \
  -C /path/to/phlex \
  config/ data/metadata/ data/plugins/ \
  --exclude='data/transcode/*' \
  --exclude='*.log'
```

#### 2.2 Built-in Phlex CLI
```bash
php bin/phlex backup:create --output /backups/phlex-$(date +%Y%m%d).tar.gz
```

#### 2.3 Automated (Cron / systemd timer)
```bash
# /etc/cron.d/phlex-backup — daily at 03:00
0 3 * * * root /usr/bin/php /home/phlex/bin/phlex backup:create \
  --output /backups/phlex-$(date +\%Y\%m\%d).tar.gz && \
  /usr/bin/find /backups -name 'phlex-*.tar.gz' -mtime +7 -delete
```

#### 2.4 Third-Party Tools (Restic / Borg / rsync)
Brief note: wrap the tarball or point rsync/restic at the above paths.
Off-site: `rclone copy` to NAS, S3, B2, or any cloud storage.

### §3 — Restore Process

Numbered step-by-step:

1. **Stop Phlex** — `systemctl stop phlex` or `php bin/phlex stop`
2. **Restore database** — `mysql -u phlex -p phlex_db < phlex-db-20250601.sql`
3. **Restore config and data** — `tar -xzf phlex-backup-20250601.tar.gz -C /`
   (verify paths match your installation)
4. **Start Phlex** — `systemctl start phlex` or `php bin/phlex start`
5. **Verify** — log in, check library items visible, confirm watch history present.
   If metadata images are missing, trigger a library rescan:
   ```bash
   php bin/phlex library:rescan --force
   ```

### §4 — Docker / docker-compose Backup

```bash
# Inside the container (backups land in the named volume)
docker compose exec phlex-server phlex backup:create \
  --output /backups/phlex-$(date +%Y%m%d).tar.gz

# Backup the named volumes directly
docker run --rm \
  -v phlex_data:/data \
  -v $(pwd):/backup \
  alpine \
  tar czf /backup/phlex_data.tar.gz -C /data .

# Restore
docker run --rm \
  -v phlex_data:/data \
  -v $(pwd):/backup \
  alpine \
  tar xzf /backup/phlex_data.tar.gz -C /data
docker compose up -d phlex-server
```

### §5 — What Can Go Wrong

#### Failure 1 — Corrupt or Incomplete Backup
**Symptom:** Restore succeeds but some tables are empty or SQL errors on import.
**Fix:** Always verify checksums after backup:
```bash
sha256sum phlex-backup-20250601.tar.gz
# Store the checksum alongside the backup or in a separate log
```
Test restores periodically (e.g., monthly) in a staging environment.

#### Failure 2 — JWT Secret Mismatch (Users Can't Log In)
**Symptom:** Database restored, but all login attempts fail with auth errors.
**Cause:** `JWT_SECRET` in `config/server.php` was not restored, or the backup
config used a different secret than the one the server is currently using.
**Fix:** Restore `config/server.php` from the same backup as the database.
Never rotate `JWT_SECRET` without also backing up the old value and
all existing refresh tokens (stored hashed in the DB).

#### Failure 3 — Media Not Re-Scanning After Restore (Stale mtime)
**Symptom:** Library shows items but poster/fanart missing; rescan doesn't
re-download metadata.
**Cause:** `FolderWatcher` uses mtime as its checksum. If the backup was
restored to the same filesystem paths, mtimes are preserved and the
watcher sees "no change."
**Fix:** Force a full rescan after restore:
```bash
php bin/phlex library:rescan --force
# OR touch all media files to update mtime:
find /path/to/media -type f -exec touch {} \;
```

#### Failure 4 — Backup Too Large (transcoded cache included)
**Symptom:** Backups are 50 GB+, slow to transfer, fill up backup storage.
**Cause:** `data/transcode/` directory included in tarball.
**Fix:** Always `--exclude='data/transcode/*'` from tar commands. Active
transcode job state is not recoverable anyway — excluding it saves space
with zero data loss.
