# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[gotcha]** PHP-DI 7's compiled-container mode (`ContainerBuilder::enableCompilation()`) rejects any closure that captures variables via `use` — `Cannot compile closures which import variables using the use keyword`. The current `Phlex\Common\Container\Providers\*` rely on `factory(static fn() use (...) => ...)`, so `PHLEX_CONTAINER_COMPILE=1` will throw at boot. Leave it unset, or refactor providers to invokable classes before flipping the flag.
- **[gotcha]** `./vendor/bin/phpunit --coverage-text` always summarises the whole `<include>` set (~6300 lines / 64 classes), NOT a filter target — there is no per-directory text mode. To verify the ≥85% bar on a narrow path (e.g. `src/Common/Container/`), parse `coverage.xml` (Clover) with `simplexml_load_file` + xpath over `//file[contains(@name, ...)]` and sum `coveredstatements / statements` per class.
- **[gotcha]** `composer require --no-scripts` still runs `optimize-autoloader` which logs `Class ... does not comply with psr-4 autoloading standard (rule: Phlex\Tests\ => ./tests). Skipping.` for many legacy test files. These warnings are pre-existing and benign — do NOT try to fix them as part of an unrelated change.
- **[gotcha]** After running `./vendor/bin/phpcbf --standard=PSR12 <file>`, the file is modified on disk and the harness emits a `staleReadFileStateHint` — re-Read it before any subsequent Edit or the Edit will fail.
- **[gotcha]** PHPUnit 10.5.63 in this repo emits one persistent deprecation: `Your XML configuration validates against a deprecated schema. Migrate ... using "--migrate-configuration"!`. It's surfaced only with `--display-all-issues` (not `--display-deprecations`). Don't chase it as a test failure — it's a known config-schema migration debt.
- **[gotcha]** `phpunit.xml` excludes `src/Server/Core/Application.php` from coverage — adding container-bootstrap tests for it won't move the coverage number. To exercise it, write an integration test under `tests/integration/Container/` (the Unit suite alone won't pick it up; a new `<testsuite name="Integration">` must be added too).
- **[convention]** Default `.gitignore` does NOT exclude `.phpunit.cache/`, `coverage-report/`, or `coverage.xml` — and `coverage.xml` is tracked in git, so every `phpunit` run dirties the tree. Step A.1 added these to `.gitignore` along with `var/cache/` (PHP-DI compile dir). Stage by path, not `git add -A`, until that change is in master.
- **[pattern]** To inject a mocked `Workerman\MySQL\Connection` into the PSR-11 container in tests, append a one-off provider AFTER `ContainerFactory::defaultProviders()` that rebinds `Connection::class => factory(static fn() => $mock)`, then call `ContainerFactory::create($config, $providers)`. PHP-DI's last-registered-wins semantics make the mock override the real-DB factory without touching production code.
- **[env]** `JWT_SECRET` env var defaults to the literal `default-secret-change-me` in `AuthServicesProvider::DEFAULT_JWT_SECRET` — preserved deliberately for parity with the pre-DI `public/index.php` bootstrap. Don't change the default; deployments are expected to set the env var.
- **[gotcha]** PHPUnit 10.5.63 in this repo emits one persistent deprecation: `Your XML configuration validates against a deprecated schema. Migrate ... using "--migrate-configuration"!`. It's surfaced only with `--display-all-issues` (not `--display-deprecations`). Don't chase it as a test failure — it's a known config-schema migration debt. None of `--display-deprecations`, `--display-warnings`, `--display-notices`, or `--display-all-issues` will print the message *body* either — they only update the trailing tally. To confirm the deprecation is the schema one (and nothing new), `git stash` your changes and re-run; the same count should persist on the base commit.
- **[gotcha]** `phpunit.xml` excludes `src/Server/Core/Application.php` from coverage — adding container-bootstrap tests for it won't move the coverage number. To exercise it, write an integration test under `tests/integration/Container/` (the Unit suite alone won't pick it up; a new `<testsuite name="Integration">` must be added too). The Integration suite exists on master as of A.1, but always invoke it explicitly with `./vendor/bin/phpunit --testsuite Integration` — the default `./vendor/bin/phpunit` runs `Unit` only.
- **[gotcha]** When adding a new constructor parameter to a service that's wired through a `*ServicesProvider` (e.g., adding `?EventDispatcherInterface $eventDispatcher` to `PlaybackController`, `MediaScanner`, `AuthManager`), the parameter is silently `null` in production unless you also add `->constructorParameter('eventDispatcher', get(EventDispatcherInterface::class))` in the matching provider (`SessionServicesProvider`, `MediaServicesProvider`, `AuthServicesProvider`). PHP-DI auto-wires by type only when there's no explicit override for the constructor; pre-existing `constructorParameter('logger', ...)` clauses make PHP-DI treat the constructor as "manually wired" for new optional params and they default to null. Unit tests that pass mocks directly won't catch this — only the integration test that drives the real container will. The DispatchSmokeTest pattern at `tests/integration/Events/DispatchSmokeTest.php` is the canonical guard.
- **[gotcha]** PHPStan level 9 rejects `Monolog\Level::fromValue($int)` with `Parameter #1 $value of static method Monolog\Level::fromValue() expects 100|200|250|300|400|500|550|600, int given.` because the method signature uses a literal int union. Fix by replacing the call with an explicit `match ($level)` that maps each `Level::*->value` constant to its enum case (with a `Level::Debug` default), rather than chasing the type with a cast or `@phpstan-ignore`.
- **[gotcha]** `phpstan-baseline.neon` entries become stale when you change the underlying code. After modifying a class that has baseline entries (e.g., `MediaScanner::$db is never read, only written` no longer applies once the constructor uses `$db` for a new query), PHPStan errors with `Ignored error pattern ... was not matched in reported errors.` Grep `phpstan-baseline.neon` for the class and delete obsolete `- message: ... path: ...` blocks rather than re-running `--generate-baseline` (which would also re-baseline genuinely new errors).
- **[pattern]** Scope `./vendor/bin/phpcs --standard=PSR12 ...` to the files YOU touched — running it across `src/` surfaces hundreds of pre-existing PSR-12 violations (whitespace, line length in `Dlna/`, etc.) unrelated to your change. List the touched paths explicitly. The minimum bar in `plans/expansion/` step files asks for clean phpcs on *new and modified files*, not the whole tree.
- **[gotcha]** `src/Common/Logger/LogChannels.php` historically defined BOTH the `LogChannels` class and the `HasLogger` trait in one file, which trips PSR-12's `Each class must be in a file by itself` sniff. As of step A.2 the trait was extracted to `src/Common/Logger/HasLogger.php`. If you reintroduce a second class/trait into a `LogChannels.php`-style file, phpcs will fail even though it parses fine.
- **[pattern]** When you need to compare a current state against the base commit (e.g., "did my change introduce this warning or was it already there?"), `git stash` → re-run the command → `git stash pop`. After popping, `git status --short` to verify untracked files came back — `stash pop` reports them in the stderr summary but doesn't error if they were already present.
- **[gotcha]** `Read` on `PHLEX_EXPANSION_PLAN.md` (65 KB / 1015 lines) returns empty `{}` when called with no `offset`/`limit`. The file is intact — pass an explicit `limit` (e.g. `limit: 350`) and read in chunks, or fall back to `Bash` with `sed -n 'A,Bp'`.
- **[gotcha]** `coverage.xml` is tracked in git (`git ls-files coverage.xml` returns it) AND listed in `.gitignore`, AND auto-regenerated by every `./vendor/bin/phpunit` run. Result: every test run dirties the tree with ` M coverage.xml`. To stage cleanly, either run `git checkout -- coverage.xml` first or stage by explicit paths (`git add phpstan-baseline.neon src/`) and never `git add -A`. PHPUnit also writes `.phpunit.cache/` and `coverage-report/` to repo root — neither is in `.gitignore` until A.1's update lands.

Output line `caliber: NN/25 events recorded — need more before analysis` during `git commit` is normal, not an error. A `caliber: skipping finalize — another finalize is in progress` line is also benign and does not block the commit.

- **[gotcha]** When you EXTRACT a class to a new file (e.g. moving `TransportState` out of `src/Dlna/AvTransport.php` into its own `src/Dlna/TransportState.php`), the baseline entry's `message:` still matches but its `path:` field now points to the *wrong* file. PHPStan reports a *double* error: the original entry as `Ignored error pattern ... was not matched in reported errors` (at the old path) AND the same error reappearing at the new path. Fix: edit the existing baseline entry's `path:` to the new file rather than regenerating the whole baseline.
- **[pattern]** Scope `./vendor/bin/phpcs --standard=PSR12 ...` to the files YOU touched when running locally — bulk runs surface hundreds of pre-existing violations unrelated to your change. BUT CI's `PHP CodeSniffer` workflow runs `./vendor/bin/phpcs --standard=PSR12 -n --colors src/` against the *whole* tree (see `.github/workflows/*`), so a PR with clean new files can still fail CI on pre-existing PSR-12 errors in unrelated modules (DLNA, LiveTv, SyncPlay). Before pushing a feature PR, run a full-tree phpcs once: if it now passes because of your changes great, otherwise either fix the bulk violations on a separate `chore-psr12-baseline` branch first or expect the CI check to be red.
- **[gotcha]** `./vendor/bin/phpcbf --standard=PSR12 src/` will auto-fix ~267 violations across ~41 files in one shot — including PSR-12's "one class per file" rule, which means it can split files (e.g. `AvTransport.php` containing both `AvTransport` and `TransportState`) by leaving the second class orphaned and requiring a manual extract to a sibling file. After phpcbf, run `php -l` on every touched file before staging — silent bracket damage can occur on multi-class files.
