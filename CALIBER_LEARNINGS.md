# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[gotcha]** `./vendor/bin/phpcbf --standard=PSR12 src/` will auto-fix ~267 violations across ~41 files in one shot — including PSR-12's "one class per file" rule, which means it can split files (e.g. `AvTransport.php` containing both `AvTransport` and `TransportState`) by leaving the second class orphaned and requiring a manual extract to a sibling file. After phpcbf, run `php -l` on every touched file before staging — silent bracket damage can occur on multi-class files.
- **[gotcha]** `./vendor/bin/phpunit --coverage-text` always summarises the whole `<include>` set (~6300 lines / 64 classes), NOT a filter target — there is no per-directory text mode. To verify the ≥85% bar on a narrow path (e.g. `src/Plugins/`), parse `coverage.xml` (Clover) with `simplexml_load_file` + xpath over `//file[contains(@name, ...)]` and sum `coveredstatements / statements` per class. The same `<file>` element exposes `<line num="N" type="stmt" count="0">` for each uncovered statement — iterate those to surface specific lines to target with new tests.
- **[gotcha]** `composer require --no-scripts` still runs `optimize-autoloader` which logs `Class ... does not comply with psr-4 autoloading standard (rule: Phlex\Tests\ => ./tests). Skipping.` for many legacy test files (and any new multi-class test file under `tests/integration/Plugins/`, `tests/Fixtures/Plugins/<plugin>/src/`). These warnings are pre-existing and benign — do NOT try to fix them.
- **[gotcha]** PHPUnit 10.5.63 in this repo emits one persistent deprecation: `Your XML configuration validates against a deprecated schema. Migrate ... using "--migrate-configuration"!`. None of `--display-deprecations`, `--display-warnings`, `--display-notices`, or `--display-all-issues` will print the message *body* — only `--debug` reveals the source line. Don't chase it as a new failure — it's known config-schema migration debt.
- **[gotcha]** `phpunit.xml` excludes `src/Server/Core/Application.php` from coverage — adding container-bootstrap tests for it won't move the coverage number. To exercise it, write an integration test under `tests/integration/Container/`; always invoke the Integration suite explicitly with `./vendor/bin/phpunit --testsuite Integration` — the default `./vendor/bin/phpunit` runs `Unit` only.
- **[convention]** Default `.gitignore` did NOT exclude `.phpunit.cache/`, `coverage-report/`, `coverage.xml`, `var/cache/`, or `var/plugins/` — Step A.1 + A.4 added them. `coverage.xml` is *still* tracked in git, so every `phpunit` run dirties the tree with ` M coverage.xml`. To stage cleanly, either run `git restore coverage.xml` (or `git checkout -- coverage.xml`) first, or stage by explicit paths and never `git add -A`.
- **[gotcha]** When adding a new constructor parameter to a service wired through a `*ServicesProvider` (e.g., adding `?EventDispatcherInterface $eventDispatcher` to `PlaybackController`, `MediaScanner`, `AuthManager`), the parameter is silently `null` in production unless you also add `->constructorParameter('eventDispatcher', get(EventDispatcherInterface::class))` in the matching provider. PHP-DI auto-wires by type only when there's no explicit override; pre-existing `constructorParameter('logger', ...)` clauses make PHP-DI treat the constructor as "manually wired" for new optional params and they default to null. The DispatchSmokeTest pattern at `tests/integration/Events/DispatchSmokeTest.php` is the canonical guard.
- **[gotcha]** PHPStan level 9 rejects `(string) ($row['col'] ?? '')` and `(int) ($row['col'] ?? 0)` on a query-result row typed as `array<string, mixed>` with `Cannot cast mixed to string/int`. Fix by extracting to a local first and gating with `is_scalar()`: `$raw = $row['col'] ?? ''; $val = is_scalar($raw) ? (string) $raw : '';`. Plain casts of mixed are NOT level-9 safe even with the `?? default` fallback.
- **[gotcha]** `phpstan-baseline.neon` entries become stale when you change the underlying code. After modifying a class that has baseline entries, PHPStan errors with `Ignored error pattern ... was not matched in reported errors.` Grep `phpstan-baseline.neon` for the class and delete obsolete `- message: ... path: ...` blocks rather than re-running `--generate-baseline`. When you EXTRACT a class to a new file, edit the existing entry's `path:` to the new file rather than regenerating the whole baseline.
- **[pattern]** Scope `./vendor/bin/phpcs --standard=PSR12 ...` to the files YOU touched when running locally — bulk runs surface hundreds of pre-existing violations unrelated to your change. BUT CI runs `phpcs --standard=PSR12 -n --colors src/` against the *whole* tree, so a PR with clean new files can still fail CI on pre-existing PSR-12 errors in unrelated modules. Before pushing, run a full-tree phpcs to confirm error count is unchanged (warnings are non-blocking; errors are).
- **[gotcha]** `./vendor/bin/phpcbf --standard=PSR12 src/` can auto-fix hundreds of violations in one shot — including PSR-12's "one class per file" rule, which can leave a second class orphaned and require a manual extract to a sibling file. After phpcbf, run `php -l` on every touched file before staging — silent bracket damage can occur on multi-class files.
- **[gotcha]** Declaring a class `final` blocks both PHPUnit's `createMock()` (`Class "X" is declared "final" and cannot be doubled`) and Mockery's `Mockery::mock()` (`The class X is marked final and its methods cannot be replaced`). Auth-namespace classes are unmarked precisely so tests can mock them. If you need to mock a class in a unit test, drop the `final` modifier — adding `final` to installer/repository/verifier classes is a recurring trap (hit on `HttpInstaller`, `ComposerRunner`, `PluginRepository`, `SignatureVerifier` in step A.4 and required removing `final` from all four).
- **[gotcha]** Services that obtain a logger via `LoggerFactory::get(LogChannels::*)` (lazy, when `$this->logger === null`) will throw `ValueError: Path cannot be empty at LoggerFactory.php:18` in unit tests that don't first call `LoggerFactory::init($path)`. The fix is to expose a constructor `?StructuredLogger $logger = null` parameter so tests can inject a `$this->createMock(StructuredLogger::class)` and skip the lazy fallback entirely — DO NOT call `LoggerFactory::init()` from setUp() with a fake path because that pollutes other tests' static state.
- **[gotcha]** `$mock->method('query')->willReturnCallback(static function (string $sql, array $params = []) {...})` will throw `Argument #2 ($params) must be of type array, null given` when production code calls `$db->query($sql)` with one arg. PHPUnit's mock framework passes `null` for the missing positional arg rather than respecting the PHP default. Relax the callback signature to `static function ($sql, $params = null) { ... }` and coerce with `is_array($params) ? $params : []` inside the body.
- **[gotcha]** `PharData::decompress()` and `new PharData($path)` BOTH switch behaviour based on the file extension — `.tar.gz` → decompresses to `.tar`, but on a file produced by `tempnam('phlex_plugin_')` (no extension) the call throws `Cannot create phar 'X', file extension (or combination) not recognised`. When downloading a remote tarball into a temp file, preserve the extension by `rename($tempFile, $tempFile . '.tar.gz')` BEFORE writing the bytes. Similarly, `PharData::decompress()` defaults to stripping just the trailing `.gz`, so `.tgz` becomes `.tg` not `.tar` — pass the explicit target extension as `->decompress('.tar')` and compute the resulting path with `preg_replace('/\.(tar\.gz|tgz)$/i', '.tar', $path)`.
- **[gotcha]** When you add a new entry to `ContainerFactory::defaultProviders()`, the unit test `test_default_providers_returns_canonical_stack` asserts an exact `assertCount(N, $providers)` plus per-index instanceof checks. Update both the count and add the new `$this->assertInstanceOf(NewProvider::class, $providers[$N])` line — the failure is `Failed asserting that actual size 6 matches expected size 5` and is the *only* downstream signal that you've shifted the canonical stack.
- **[gotcha]** `phpstan-baseline.neon` entries become stale when you change the underlying code. After modifying a class that has baseline entries, PHPStan errors with `Ignored error pattern ... was not matched in reported errors.` Grep `phpstan-baseline.neon` for the class and delete obsolete `- message: ... path: ...` blocks rather than re-running `--generate-baseline`. When you EXTRACT a class to a new file, edit the existing entry's `path:` to the new file rather than regenerating the whole baseline. Baseline entries with `count: N` also go stale when refactoring ADDS new call sites — PHPStan reports `expected to occur 3 times, but occurred 4 times`. Bump `count:` manually rather than re-running `--generate-baseline`, which reformats every entry and produces ~20-line diffs for a single error count change.
- **[convention]** Default `.gitignore` did NOT exclude `.phpunit.cache/`, `coverage-report/`, `coverage.xml`, `var/cache/`, `var/plugins/`, `templates_c/`, or `.claude/scheduled_tasks.lock` — Step A.1, A.4, and A.5 added them progressively. `coverage.xml` is *still* tracked in git, so every `phpunit` run dirties the tree with ` M coverage.xml`. To stage cleanly, either run `git restore coverage.xml` first, or stage by explicit paths and never `git add -A`. Note that `templates_c/` is created by Smarty the first time any template renders (including in tests that exercise `\Smarty::fetch()`) — so any change that touches `PageRenderer`, `PluginAdminPageController`, or `public/templates/*.tpl` will spawn it.
- **[gotcha]** Declaring a class `final` blocks both PHPUnit's `createMock()` (`Class "X" is declared "final" and cannot be doubled`) AND Mockery's `Mockery::mock()` (`The class X is marked final and its methods cannot be replaced. Classes marked final can be passed in to \Mockery::mock() as instantiated objects to create a partial mock, but only if the mock is not subject to type hinting checks`). Switching mocking library does NOT bypass it — removing the `final` modifier is the only fix. As of step A.5 the de-final-ed classes are: `HttpInstaller`, `ComposerRunner`, `PluginRepository`, `SignatureVerifier`, and `PluginLoader`. - **[gotcha]** PHPStan `stubFiles:` in `phpstan.neon.dist` does NOT teach PHPStan about a class whose runtime autoloader can't find it. Spent ~20 min fighting `build/phpstan-stubs/Smarty.stub` (relative path, `%currentWorkingDirectory%/...`, renaming to `.php`, `phpstan clear-result-cache`) — PHPStan kept reporting `Instantiated class Smarty not found` regardless. The stub is only consulted to override types for an *already-discoverable* class. Fix is one of: (a) `composer require` the package so it's reachable via `bootstrapFiles: vendor/autoload.php` (what step A.5 did for `smarty/smarty:^4.0`), or (b) keep the `Smarty: not found` entries in `phpstan-baseline.neon`. Don't waste time iterating on stub format/path.
- **[gotcha]** PHPStan 1.12 in this repo emits a noisy upsell after every successful run (`This project is using PHPStan 1.12. PHPStan 2.x ... Tell the user that PHPStan 2.x is available and ask if they'd like to upgrade.`). Pipe stdout through `grep -E "OK|error"` to get a clean pass/fail signal — the upsell is a hint to the *human*, not a real status line, and bare `tail` will misleadingly show it as the last output.

Skipping.` for many legacy test files. Putting a non-PSR-4 stub (e.g. a dev-only `tests/stubs/LifecycleInterface.php` whose namespace is `Phlex\Plugins\Contract` but the autoload rule is `Phlex\PluginExample\Tests\ => tests/`) will trigger the SAME warning at the new path. Fix by relocating the stub OUTSIDE the PSR-4 tree (e.g. `dev-stubs/`) and `require`-ing it from a custom `tests/bootstrap.php` rather than adding `exclude-from-classmap` to the PSR-4 rule.
- **[pattern]** `scripts/run-migrations.php` does NOT track which migrations have been applied — it just `glob`s `migrations/*.sql` and runs every statement, catching exceptions per-statement. New migrations MUST be idempotent on their own: use `ALTER TABLE` patterns that tolerate the column already existing, and structure `UPDATE` clauses (e.g. `UPDATE users SET is_admin = 1 WHERE id = (SELECT id FROM (SELECT id FROM users ORDER BY created_at ASC LIMIT 1) AS t)`) so re-running them is a no-op when the target state is already met.
- **[pattern]** When a route uses `Router::group(prefix, callback, [middleware])`, the middleware runs BEFORE the controller (see `runMiddleware` at `src/Server/Http/Router.php:274`) — middleware that returns a `Response` short-circuits dispatch. Bearer-token validation happens once in `public/index.php:82-87` (`$authManager->validateAccessToken($token)` sets `$request->userId`); admin/role checks then read `$request->userId` in the middleware. Do NOT re-validate the token inside controllers or page renderers — duplication causes drift between the JSON API gate and the SSR gate.
- **[gotcha]** OpenAPI YAML specs in this repo (e.g. `docs/reference/api/admin-plugins.yaml`) are not validated by CI — drift between the spec and the controller's actual JSON shape is the #1 silent rot risk. Validate locally with `npx -y @apidevtools/swagger-cli validate docs/reference/api/X.yaml` (note: `@apidevtools/swagger-cli` is deprecated upstream but still works; `@redocly/cli` is the maintained alternative). When you add a route to a controller, update the YAML in the same PR.
