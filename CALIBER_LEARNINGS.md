- **[fix]** `./vendor/bin/phpstan` and `./vendor/bin/phpcs` ARE now installed as of step A.1. Config lives at `phpstan.neon.dist` + `phpstan-baseline.neon` (baseline absorbs ~823 pre-existing errors so new code is held to level 9 without a repo-wide refactor).
- **[gotcha]** `./vendor/bin/phpunit --coverage-text` always summarises the whole `<include>` set (~6300 lines / 64 classes), NOT a filter target — there is no per-directory text mode. To verify the ≥85% bar on a narrow path (e.g. `src/Plugins/`), parse `coverage.xml` (Clover) with `grep -A1 "src/Plugins/Manifest.php" coverage.xml` or `simplexml_load_file` + xpath over `//file[contains(@name, ...)]` and read each class's `<metrics ... coveredstatements="N" statements="M"/>` attributes — those let you compute per-class coverage without a custom test run.
- **[gotcha]** PHPUnit 10.5.63 in this repo emits one persistent deprecation: `Your XML configuration validates against a deprecated schema. Migrate ... using "--migrate-configuration"!`. None of `--display-deprecations`, `--display-warnings`, `--display-notices`, or `--display-all-issues` will print the message *body* — only `--debug` reveals the source line `Test Runner Triggered Deprecation (Your XML configuration validates against a deprecated schema...)`. Don't chase it as a new failure — it's a known config-schema migration debt.
- **[gotcha]** `phpunit.xml` excludes `src/Server/Core/Application.php` from coverage — adding container-bootstrap tests for it won't move the coverage number. To exercise it, write an integration test under `tests/integration/Container/`; the Integration suite exists on master as of A.1, but always invoke it explicitly with `./vendor/bin/phpunit --testsuite Integration` — the default `./vendor/bin/phpunit` runs `Unit` only.
- **[gotcha]** When adding a new constructor parameter to a service that's wired through a `*ServicesProvider` (e.g., adding `?EventDispatcherInterface $eventDispatcher` to `PlaybackController`, `MediaScanner`, `AuthManager`), the parameter is silently `null` in production unless you also add `->constructorParameter('eventDispatcher', get(EventDispatcherInterface::class))` in the matching provider. PHP-DI auto-wires by type only when there's no explicit override; pre-existing `constructorParameter('logger', ...)` clauses make PHP-DI treat the constructor as "manually wired" for new optional params and they default to null. The DispatchSmokeTest pattern at `tests/integration/Events/DispatchSmokeTest.php` is the canonical guard.
- **[gotcha]** PHPStan level 9 rejects `Monolog\Level::fromValue($int)` with `Parameter #1 $value of static method Monolog\Level::fromValue() expects 100|200|250|300|400|500|550|600, int given.` because the method signature uses a literal int union. Fix by replacing the call with an explicit `match ($level)` that maps each `Level::*->value` constant to its enum case (with a `Level::Debug` default).
- **[gotcha]** `phpstan-baseline.neon` entries become stale when you change the underlying code. After modifying a class that has baseline entries (e.g., `MediaScanner::$db is never read, only written` no longer applies once the constructor uses `$db` for a new query), PHPStan errors with `Ignored error pattern ... was not matched in reported errors.` Grep `phpstan-baseline.neon` for the class and delete obsolete `- message: ... path: ...` blocks rather than re-running `--generate-baseline`.
- **[gotcha]** When you EXTRACT a class to a new file (e.g. moving `TransportState` out of `src/Dlna/AvTransport.php` into its own `src/Dlna/TransportState.php`), the baseline entry's `message:` still matches but its `path:` field now points to the *wrong* file. PHPStan reports a *double* error: the original entry as `Ignored error pattern ... was not matched in reported errors` (at the old path) AND the same error reappearing at the new path. Fix: edit the existing baseline entry's `path:` to the new file rather than regenerating the whole baseline.
- **[pattern]** Scope `./vendor/bin/phpcs --standard=PSR12 ...` to the files YOU touched when running locally — bulk runs surface hundreds of pre-existing violations unrelated to your change. BUT CI's `PHP CodeSniffer` workflow runs `./vendor/bin/phpcs --standard=PSR12 -n --colors src/` against the *whole* tree (see `.github/workflows/*`), so a PR with clean new files can still fail CI on pre-existing PSR-12 errors in unrelated modules (DLNA, LiveTv, SyncPlay). Before pushing a feature PR, run a full-tree phpcs once: if it now passes because of your changes great, otherwise either fix the bulk violations on a separate `chore-psr12-baseline` branch first or expect the CI check to be red.
- **[gotcha]** `./vendor/bin/phpcbf --standard=PSR12 src/` will auto-fix ~267 violations across ~41 files in one shot — including PSR-12's "one class per file" rule, which means it can split files (e.g. `AvTransport.php` containing both `AvTransport` and `TransportState`) by leaving the second class orphaned and requiring a manual extract to a sibling file. After phpcbf, run `php -l` on every touched file before staging — silent bracket damage can occur on multi-class files.

- **[gotcha]** PHPStan level 9 rejects `(string) ($row['col'] ?? '')` and `(int) ($row['col'] ?? 0)` on a query-result row typed as `array<string, mixed>` with `Cannot cast mixed to string/int`. - **[gotcha]** Declaring a class `final` blocks both PHPUnit's `createMock()` (`Class "X" is declared "final" and cannot be doubled`) and Mockery's `Mockery::mock()` (`The class X is marked final and its methods cannot be replaced`). Auth-namespace classes are unmarked precisely so tests can mock them. If you need to mock a class in a unit test, drop the `final` modifier — adding `final` to installer/repository/verifier classes is a recurring trap (hit on `HttpInstaller`, `ComposerRunner`, `PluginRepository`, `SignatureVerifier` in step A.4 and required removing `final` from all four).
- **[gotcha]** Services that obtain a logger via `LoggerFactory::get(LogChannels::*)` (lazy, when `$this->logger === null`) will throw `ValueError: Path cannot be empty at LoggerFactory.php:18` in unit tests that don't first call `LoggerFactory::init($path)`. The fix is to expose a constructor `?StructuredLogger $logger = null` parameter so tests can inject a `$this->createMock(StructuredLogger::class)` and skip the lazy fallback entirely — DO NOT call `LoggerFactory::init()` from setUp() with a fake path because that pollutes other tests' static state.
- **[gotcha]** `$mock->method('query')->willReturnCallback(static function (string $sql, array $params = []) {...})` will throw `Argument #2 ($params) must be of type array, null given` when production code calls `$db->query($sql)` with one arg. PHPUnit's mock framework passes `null` for the missing positional arg rather than respecting the PHP default. Relax the callback signature to `static function ($sql, $params = null) { ... }` and coerce with `is_array($params) ? $params : []` inside the body.
- **[gotcha]** `PharData::decompress()` and `new PharData($path)` BOTH switch behaviour based on the file extension — `.tar.gz` → decompresses to `.tar`, but on a file produced by `tempnam('phlex_plugin_')` (no extension) the call throws `Cannot create phar 'X', file extension (or combination) not recognised`. When downloading a remote tarball into a temp file, preserve the extension by `rename($tempFile, $tempFile . '.tar.gz')` BEFORE writing the bytes. Similarly, `PharData::decompress()` defaults to stripping just the trailing `.gz`, so `.tgz` becomes `.tg` not `.tar` — pass the explicit target extension as `->decompress('.tar')` and compute the resulting path with `preg_replace('/\.(tar\.gz|tgz)$/i', '.tar', $path)`.
- **[gotcha]** When you add a new entry to `ContainerFactory::defaultProviders()`, the unit test `test_default_providers_returns_canonical_stack` asserts an exact `assertCount(N, $providers)` plus per-index instanceof checks. Update both the count and add the new `$this->assertInstanceOf(NewProvider::class, $providers[$N])` line — the failure is `Failed asserting that actual size 6 matches expected size 5` and is the *only* downstream signal that you've shifted the canonical stack.
- **[gotcha]** `phpstan-baseline.neon` entries become stale when you change the underlying code. After modifying a class that has baseline entries, PHPStan errors with `Ignored error pattern ... was not matched in reported errors.` Grep `phpstan-baseline.neon` for the class and delete obsolete `- message: ... path: ...` blocks rather than re-running `--generate-baseline`. Bump `count:` manually rather than re-running `--generate-baseline`, which reformats every entry and produces ~20-line diffs for a single error count change.

- **[gotcha]** PHPStan `stubFiles:` in `phpstan.neon.dist` does NOT teach PHPStan about a class whose runtime autoloader can't find it. Spent ~20 min fighting `build/phpstan-stubs/Smarty.stub` (relative path, `%currentWorkingDirectory%/...`, renaming to `.php`, `phpstan clear-result-cache`) — PHPStan kept reporting `Instantiated class Smarty not found` regardless. The stub is only consulted to override types for an *already-discoverable* class. Fix is one of: (a) `composer require` the package so it's reachable via `bootstrapFiles: vendor/autoload.php` (what step A.5 did for `smarty/smarty:^4.0`), or (b) keep the `Smarty: not found` entries in `phpstan-baseline.neon`. Don't waste time iterating on stub format/path.
- **[gotcha]** PHPStan 1.12 in this repo emits a noisy upsell after every successful run (`This project is using PHPStan 1.12. PHPStan 2.x ... Tell the user that PHPStan 2.x is available and ask if they'd like to upgrade.`). Pipe stdout through `grep -E "OK|error"` to get a clean pass/fail signal — the upsell is a hint to the *human*, not a real status line, and bare `tail` will misleadingly show it as the last output.
