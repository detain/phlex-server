# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[gotcha]** PHPStan level 9 rejects `(string) ($row['col'] ?? '')` and `(int) ($row['col'] ?? 0)` on a query-result row typed as `array<string, mixed>` with `Cannot cast mixed to string/int`. Fix by extracting to a local first and gating with `is_scalar()`: `$raw = $row['col'] ?? ''; $val = is_scalar($raw) ? (string) $raw : '';`. Plain casts of mixed are NOT level-9 safe even with the `?? default` fallback.
- **[gotcha]** `phpstan-baseline.neon` entries become stale when you change the underlying code. After modifying a class that has baseline entries, PHPStan errors with `Ignored error pattern ... was not matched in reported errors.` Grep `phpstan-baseline.neon` for the class and delete obsolete `- message: ... path: ...` blocks rather than re-running `--generate-baseline`. When you EXTRACT a class to a new file, edit the existing entry's `path:` to the new file rather than regenerating the whole baseline.
- **[pattern]** Scope `./vendor/bin/phpcs --standard=PSR12 ...` to the files YOU touched when running locally — bulk runs surface hundreds of pre-existing violations unrelated to your change. BUT CI runs `phpcs --standard=PSR12 -n --colors src/` against the *whole* tree, so a PR with clean new files can still fail CI on pre-existing PSR-12 errors in unrelated modules. Before pushing, run a full-tree phpcs to confirm error count is unchanged (warnings are non-blocking; errors are).
- **[gotcha]** `./vendor/bin/phpcbf --standard=PSR12 src/` can auto-fix hundreds of violations in one shot — including PSR-12's "one class per file" rule, which can leave a second class orphaned and require a manual extract to a sibling file. After phpcbf, run `php -l` on every touched file before staging — silent bracket damage can occur on multi-class files.
- **[gotcha]** Declaring a class `final` blocks both PHPUnit's `createMock()` (`Class "X" is declared "final" and cannot be doubled`) and Mockery's `Mockery::mock()` (`The class X is marked final and its methods cannot be replaced`). Auth-namespace classes are unmarked precisely so tests can mock them. If you need to mock a class in a unit test, drop the `final` modifier — adding `final` to installer/repository/verifier classes is a recurring trap (hit on `HttpInstaller`, `ComposerRunner`, `PluginRepository`, `SignatureVerifier` in step A.4 and required removing `final` from all four).
- **[gotcha]** Services that obtain a logger via `LoggerFactory::get(LogChannels::*)` (lazy, when `$this->logger === null`) will throw `ValueError: Path cannot be empty at LoggerFactory.php:18` in unit tests that don't first call `LoggerFactory::init($path)`. The fix is to expose a constructor `?StructuredLogger $logger = null` parameter so tests can inject a `$this->createMock(StructuredLogger::class)` and skip the lazy fallback entirely — DO NOT call `LoggerFactory::init()` from setUp() with a fake path because that pollutes other tests' static state.
- **[gotcha]** `$mock->method('query')->willReturnCallback(static function (string $sql, array $params = []) {...})` will throw `Argument #2 ($params) must be of type array, null given` when production code calls `$db->query($sql)` with one arg. PHPUnit's mock framework passes `null` for the missing positional arg rather than respecting the PHP default. Relax the callback signature to `static function ($sql, $params = null) { ... }` and coerce with `is_array($params) ? $params : []` inside the body.
- **[gotcha]** `PharData::decompress()` and `new PharData($path)` BOTH switch behaviour based on the file extension — `.tar.gz` → decompresses to `.tar`, but on a file produced by `tempnam('phlex_plugin_')` (no extension) the call throws `Cannot create phar 'X', file extension (or combination) not recognised`. When downloading a remote tarball into a temp file, preserve the extension by `rename($tempFile, $tempFile . '.tar.gz')` BEFORE writing the bytes. Similarly, `PharData::decompress()` defaults to stripping just the trailing `.gz`, so `.tgz` becomes `.tg` not `.tar` — pass the explicit target extension as `->decompress('.tar')` and compute the resulting path with `preg_replace('/\.(tar\.gz|tgz)$/i', '.tar', $path)`.
- **[gotcha]** When you add a new entry to `ContainerFactory::defaultProviders()`, the unit test `test_default_providers_returns_canonical_stack` asserts an exact `assertCount(N, $providers)` plus per-index instanceof checks. Update both the count and add the new `$this->assertInstanceOf(NewProvider::class, $providers[$N])` line — the failure is `Failed asserting that actual size 6 matches expected size 5` and is the *only* downstream signal that you've shifted the canonical stack.
- **[gotcha]** `phpstan-baseline.neon` entries become stale when you change the underlying code. After modifying a class that has baseline entries, PHPStan errors with `Ignored error pattern ... was not matched in reported errors.` Grep `phpstan-baseline.neon` for the class and delete obsolete `- message: ... path: ...` blocks rather than re-running `--generate-baseline`. When you EXTRACT a class to a new file, edit the existing entry's `path:` to the new file rather than regenerating the whole baseline. Baseline entries with `count: N` also go stale when refactoring ADDS new call sites — PHPStan reports `expected to occur 3 times, but occurred 4 times`. Bump `count:` manually rather than re-running `--generate-baseline`, which reformats every entry and produces ~20-line diffs for a single error count change.
- **[convention]** Default `.gitignore` did NOT exclude `.phpunit.cache/`, `coverage-report/`, `coverage.xml`, `var/cache/`, `var/plugins/`, `templates_c/`, or `.claude/scheduled_tasks.lock` — Step A.1, A.4, and A.5 added them progressively. `coverage.xml` is *still* tracked in git, so every `phpunit` run dirties the tree with ` M coverage.xml`. To stage cleanly, either run `git restore coverage.xml` first, or stage by explicit paths and never `git add -A`. - **[gotcha]** PHPStan `stubFiles:` in `phpstan.neon.dist` does NOT teach PHPStan about a class whose runtime autoloader can't find it. Spent ~20 min fighting `build/phpstan-stubs/Smarty.stub` (relative path, `%currentWorkingDirectory%/...`, renaming to `.php`, `phpstan clear-result-cache`) — PHPStan kept reporting `Instantiated class Smarty not found` regardless. The stub is only consulted to override types for an *already-discoverable* class. Fix is one of: (a) `composer require` the package so it's reachable via `bootstrapFiles: vendor/autoload.php` (what step A.5 did for `smarty/smarty:^4.0`), or (b) keep the `Smarty: not found` entries in `phpstan-baseline.neon`. Don't waste time iterating on stub format/path.
- **[gotcha]** PHPStan 1.12 in this repo emits a noisy upsell after every successful run (`This project is using PHPStan 1.12. PHPStan 2.x ... Tell the user that PHPStan 2.x is available and ask if they'd like to upgrade.`). Pipe stdout through `grep -E "OK|error"` to get a clean pass/fail signal — the upsell is a hint to the *human*, not a real status line, and bare `tail` will misleadingly show it as the last output.
- **[gotcha]** `--coverage-text` redirected to a file (`--coverage-text=/tmp/x.txt`) writes an EMPTY file in this repo's PHPUnit setup — the report only flushes to stdout when the parameter is absent. Skipping.` for many legacy test files. - **[pattern]** `scripts/run-migrations.php` does NOT track which migrations have been applied — it just `glob`s `migrations/*.sql` and runs every statement, catching exceptions per-statement. New migrations MUST be idempotent on their own: use `ALTER TABLE` patterns that tolerate the column already existing, and structure `UPDATE` clauses (e.g. `UPDATE users SET is_admin = 1 WHERE id = (SELECT id FROM (SELECT id FROM users ORDER BY created_at ASC LIMIT 1) AS t)`) so re-running them is a no-op when the target state is already met.
- **[pattern]** When a route uses `Router::group(prefix, callback, [middleware])`, the middleware runs BEFORE the controller (see `runMiddleware` at `src/Server/Http/Router.php:274`) — middleware that returns a `Response` short-circuits dispatch. Bearer-token validation happens once in `public/index.php:82-87` (`$authManager->validateAccessToken($token)` sets `$request->userId`); admin/role checks then read `$request->userId` in the middleware. Do NOT re-validate the token inside controllers or page renderers — duplication causes drift between the JSON API gate and the SSR gate.

- **[gotcha]** PHPStan level 9 does NOT narrow a guarded nullable property across multiple statements: `if ($this->db !== null) { $this->db->beginTrans(); ... $this->db->commitTrans(); }` will report `Cannot call method commitTrans() on Workerman\MySQL\Connection|null` because PHPStan re-widens `$this->db` after every statement (a method call could mutate it). The fix is to copy the property to a local once: `$db = $this->db; if ($db !== null) { $db->beginTrans(); ... $db->commitTrans(); }`. This pattern recurs anywhere `AuthManager`, `PluginLoader`, or repositories wrap multi-step DB work in nullable-Connection transactions.

- **[convention]** `SignatureVerifier::verify($manifest, $directory)` is a two-step gate: (1) content-integrity check — `hash_file('sha256', $directory . '/plugin.json')` must equal the digest portion of the manifest's `signature` field, mismatch ALWAYS returns `RESULT_INVALID` regardless of allowlist; (2) allowlist check — only after content matches. A signed manifest with an empty allowlist + `requireSignature=false` is optimistically `RESULT_VALID`; with `requireSignature=true` it is always `RESULT_INVALID`. When testing the verifier, write actual bytes to a temp `plugin.json` whose `hash('sha256', $bytes)` matches the manifest's signature — `Manifest::fromArray()` builds a value object but does NOT read the file, so a test without a real file on disk will get `RESULT_INVALID` for the missing-file branch.
